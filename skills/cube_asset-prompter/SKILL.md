# WowCube Asset Prompter

Bridge between design (GDD) and art generation (`canvas-design`). This
skill never renders pixels and never writes game code. It produces the
prompt package that a downstream renderer will execute to fill
`assets/art/*.png` and `assets/mp3/*.mp3` with real content instead of
deterministic placeholders.

**Core principle:** every prompt is self-contained and uses an asset name
that is already guaranteed to pass `cube_asset-builder`'s validator. The
art bible, not the individual prompt, carries the shared visual language
so that N independent `canvas-design` invocations yield one coherent set.

## When to Use

- A GDD exists at `plans/<game>_gdd.md` AND a manifest exists at
  `plans/<game>_assets.json` (or `plans/<game>/<game>_assets.json`).
- The user says "generate art prompts", "prepare art", "make asset
  prompts", or similar after `technical_prompter` finished.
- Resuming after the user requested a style change — regenerate the art
  bible and the affected prompts.

## When NOT to Use

- No GDD → delegate to `cube_game-designer`.
- No manifest → delegate to `technical_prompter`.
- The user wants real PNGs on disk → this skill emits prompts only; the
  renderer (`canvas-design` per prompt) is a separate invocation.
- The user wants the packed atlas, `pal.png`, or `_ids.h` → that is
  `cube_asset-builder`'s territory.

## Execution environment

This skill is pure text I/O — it reads markdown and JSON, writes
markdown. No shell commands, no dependencies, no rendering. The prompts
it produces are later consumed by `canvas-design`, whose scripts run in
the agent's `mcp__workspace__bash` sandbox (never on the user's machine).

## Prerequisites

File inputs (must already be in the user's workspace):

| File | Source | Required |
|------|--------|----------|
| `plans/<game>_gdd.md` OR `plans/<game>/<game>_gdd.md` | `cube_game-designer` | Yes |
| `plans/<game>_assets.json` OR `plans/<game>/<game>_assets.json` | `technical_prompter` | Yes |
| `plans/<game>/style_profile.md` | this skill, Step 0 (created if missing) | Yes |
| `skills/canvas-design/canvas-fonts/Rubik-Bold.ttf` | Vendored brand font | Yes |

If any prerequisite is missing, print the gap and stop. In particular, **never invent a visual style.** If `style_profile.md` is missing, run **Step 0: Style Profile Selection** (below) BEFORE any other step — do not pick a style yourself, and do not reuse a previous game's profile.

## Workflow

### Step 0: Style Profile Selection (MANDATORY for new games)

This skill OWNS the visual style decision for the game. The choice is a product-level call that affects the GDD's "Style & References" section, the art bible, every per-asset prompt, the rendered PNGs, and even some code-side constants (BG color, palette assumptions). It must be decided **before** any art bible or per-asset prompt is written.

If `plans/<game>/style_profile.md` already exists (resuming a game, or `cube_orchestrator` rotated it later), skip to Step 1.
If it does NOT exist, do the following BEFORE any other step:

1. **List available presets** by reading `OCT_wowcube-agent-skills/skills/cube_orchestrator/styles/README.md`. The presets shipped today are:
   - `detailed_pixelart` — multi-tonal pixel-art with material identity (recommended default for arcade/action)
   - `minimalist_flat` — bold flat shapes, fastest to render, smallest palette
   - `cartoon_thick_outline` — casual cel-shaded vector look
   - `retro_8bit` — strict 16-color palette, NES/PICO-8 era
   - `painterly_storybook` — soft watercolor, narrative games
   - `realistic_render` — photo-leaning rendering, smooth gradients, ≤256 colors, no outline

2. **Ask the user** which preset they want, OR offer the recommended default with a one-line rationale based on the game genre. Use `AskUserQuestion` if available; otherwise present a short bullet list and wait for explicit reply.

3. **Copy the chosen preset** to `plans/<game>/style_profile.md`. Verbatim copy — do NOT paraphrase. Append a small header at the top recording the source preset name and the date:

   ```markdown
   <!-- style_profile generated by cube_asset-prompter -->
   <!-- source preset: detailed_pixelart -->
   <!-- created: <ISO-8601 date> -->
   <!-- game: <game_name> -->

   <preset content verbatim>
   ```

4. **Honor the profile downstream.** The art bible written in Step 4 inherits the profile's "Visual identity", "Palette policy", and "Reference language for prompts" sections verbatim. Every per-asset prompt written in Step 6 quotes 2–4 sentences from the profile's "Reference language for prompts" section. The profile is the law; the GDD is the flavor on top.

5. **Forbid procedural-fallback art (signal to downstream).** A common failure mode is to substitute the real `canvas-design` rendering with a quick PIL/SVG primitive script "for speed". When the chosen style is anything other than `minimalist_flat`, every per-asset prompt MUST explicitly forbid procedural fallbacks (under "Hard no's"). The `minimalist_flat` preset is the ONLY style where procedural primitives are acceptable, because it was designed for that. The actual enforcement happens in `canvas-design` and `cube_orchestrator`, but the prohibition is signalled in the prompts this skill writes.

6. **Style rotation belongs to `cube_orchestrator`.** If, later in the pipeline, the user asks to "change the look" or "make it more <X>", do NOT handle that here — it is `cube_orchestrator`'s job to rotate the profile, then re-invoke this skill to regenerate the art bible and per-asset prompts, then re-run `canvas-design` and `cube_asset-builder`. This skill owns the **initial** selection only; the orchestrator owns the **lifecycle**.

### Step 0 output

- `plans/<game>/style_profile.md` exists and is a verbatim copy of one preset from `OCT_wowcube-agent-skills/skills/cube_orchestrator/styles/`.

### Step 1: Locate inputs

### Step 2: Pre-flight validation of the manifest

Before writing any prompt, walk the manifest and enforce the same rules
that `cube_asset-builder`'s validator will apply later. If any rule
fails, stop and list every violation — do NOT proceed to prompt
generation. This catches errors early so the renderer does not waste
cycles on doomed names.

**Name format (sprites and sounds):**

- Regex: `^[a-z0-9_]+$` — lowercase latin, digits, underscore only.
- No uppercase, no hyphens, no spaces, no unicode, no punctuation.
- No duplicates within sprites. No duplicates within sounds.

**Reserved names (forbidden for any sprite or sound):**

- `pal`
- `0`
- `icon`
- `none`
- `bmp_none`
- `bmp_last`
- `bmp_0`
- `map_none`
- `map_last`

**Animation frame sequences:**

- Suffix is exactly two digits, zero-padded: `_00` .. `_99`. Single-digit
  (`_0`) and three-digit (`_000`) suffixes are invalid.
- Every frame of an animation MUST set both `anim: "<base>"` and
  `frame: <N>` (0-based integer). If one is present the other must be.
- The sequence MUST start at frame 0 (file suffix `_00`).
- Frames MUST be contiguous — `_00, _01, _02, ...` with no gaps.
- File name MUST equal `<anim>_<NN>` exactly, where `<NN>` is the
  zero-padded frame index. Example: `anim="hero_idle"`, `frame=0` →
  name `hero_idle_00`; `frame=1` → `hero_idle_01`.
- Derived group rule: if `group` is not set, strip the trailing `_NN`
  from the name. So `hero_idle_00` → group `hero_idle`. Use this to
  bucket all frames of one animation into a single visual style brief.

**Size and duration bounds:**

- Sprite `size` = `[w, h]` with `1 <= w <= 240` and `1 <= h <= 240`.
- Sound `duration_ms` in `1..2000` (default 500 if omitted).
- Sound `event_type` (when present) must be one of: `pickup`, `hit`,
  `ui`, `ambient`, `music`, `default`.

Failure output format (one line per error, stop after the list):

```
manifest validation failed:
  sprite 'Hero_Idle': name must match [a-z0-9_]+
  sprite 'coin_001': frame suffix must be two digits _00.._99
  anim 'enemy': sequence must start at frame 0 (found first frame 1)
  anim 'enemy': frames must be contiguous, found [0, 1, 3]
  sound 'pal': reserved name
  sprite 'boss': size [300, 300] exceeds 240 per axis
  sound 'boom': duration_ms 3000 exceeds 2000
```

### Step 3: Extract visual style — style_profile is the law, GDD is the flavor

The visual style is decided in Step 0 of THIS skill (or rotated later by `cube_orchestrator`) and lives at `plans/<game>/style_profile.md`. Read it FIRST and IN FULL — it is non-negotiable.

Then read `plans/<game>_gdd.md` for the game-specific flavor on top of the chosen style:

1. **`plans/<game>/style_profile.md`** — **PRIMARY SOURCE.** Tone count, light direction, edge treatment, dithering policy, palette structure, outline policy. The art bible inherits these rules verbatim. If the GDD contradicts the style profile, the style profile wins and the contradiction is recorded as an assumption.
2. **GDD §1.2 Style & References** — game-specific palette colors, mood words, inspiration references that don't conflict with the style profile.
3. **GDD §1.1 Overview** — one-liner that fixes the game's tone.
4. **GDD §3 Game Objects** — per-object appearance hints. Refines §1.2 for specific objects. Subordinate to the style profile.
5. **GDD §6 Screens & Feedback** — how visuals behave in motion (animation style, feedback effects). Subordinate to the style profile.

Do NOT invent style beyond what `style_profile.md` and the GDD specify. If a value is missing in both (e.g., GDD says "warm fruit colors" without hex codes), pick a concrete value compatible with the style profile's palette policy and log it as an assumption in the art bible's "Assumptions" section. Never silently make up colors that violate the style profile.

### Step 4: Write the art bible

Create `plans/<game>/art_bible.md` with the structure below. The art
bible is the single source of truth every per-asset prompt refers back
to — it is what guarantees visual consistency across the set.

```markdown
# <Game Name> — Art Bible

> One-sentence mission: what this game should feel like visually.

## 0. Style Profile (binding)

This game's visual style is **`<style_profile_name>`**, locked in by
`cube_asset-prompter` Step 0 (or rotated by `cube_orchestrator`) and
documented at [`style_profile.md`](style_profile.md). Every rule below MUST conform
to that profile's "Visual identity", "Palette policy", and "Reference
language for prompts" sections. In case of conflict between this art
bible and `style_profile.md`, the style profile wins.

Briefly summarize the binding rules from the profile here (3–5 bullets,
quoted from the profile so per-asset prompts can read them inline):
- Tone count and shading rule
- Light direction
- Edge / outline policy
- Pixel-grid policy (true pixel-art? AA permitted?)
- Palette size budget

## 1. Palette

List 4–8 colors. For each: hex code, role (primary / accent / shadow /
highlight / background), and where it is used. Derive from GDD §1.2. If
the GDD gives named colors without hex codes, pick hex values that fit
the description and mark them `(assumed)`.

| Role       | Hex       | Usage                                |
|------------|-----------|--------------------------------------|
| Primary    | `#______` | Main character body, key UI elements |
| Accent     | `#______` | Pickups, positive feedback           |
| Shadow     | `#______` | Depth, rim lighting                  |
| ...        | ...       | ...                                  |

## 2. Line & Form

- Line weight: e.g., 2px uniform / variable / none (flat shapes only).
- Edge treatment: crisp / soft / dithered.
- Corner radius philosophy: sharp / rounded / mixed.
- Silhouette complexity: simple primitives / moderate detail / dense.

## 3. Shading & Volume

- Flat / 2-tone / gradient / painterly.
- Light direction if applicable (e.g., top-left 45°).
- Ambient occlusion policy.

## 4. Level of Abstraction

Where on the abstract ↔ figurative axis. State explicitly whether
sprites are literal depictions (a coin looks like a coin) or
stylized (a coin is a yellow circle with a glyph). Game sprites at
48×48 cannot carry the same detail as a poster; err toward clarity of
silhouette.

## 5. Typography

- **Brand typeface (locked):** `skills/canvas-design/canvas-fonts/Rubik-Bold.ttf`
- **Usage:** Titles, numeric HUD, any in-game text. No other typeface
  is permitted anywhere in the asset set. System fallbacks are
  forbidden — if the file cannot be loaded, the render must fail, not
  silently substitute.
- **Case:** UPPERCASE for titles, mixed for labels.
- **Tracking / kerning:** describe only if GDD specifies.

## 6. Mood & References

Prose paragraph (3–6 sentences) capturing the game's tonal direction,
quoting the GDD §1.1 and §1.2 where possible. Name 2–4 concrete
inspirations from the GDD (other games, films, art movements). This
is the "soul" hook every per-asset prompt will gesture at.

## 7. Asset Naming Contract

Every prompt in `prompts/` references an asset file by exact name.
Those names are fixed by `plans/<game>_assets.json` and obey the
rules below. Renderers MUST save output files under these names
verbatim — no case changes, no dashes, no spaces.

- **Character set:** `^[a-z0-9_]+$` — lowercase latin, digits,
  underscore only. No uppercase, hyphens, spaces, unicode, or
  punctuation.
- **Reserved names (forbidden):** `pal`, `0`, `icon`, `none`,
  `bmp_none`, `bmp_last`, `bmp_0`, `map_none`, `map_last`.
- **Animation frames:** zero-padded two-digit suffix `_00` through
  `_99`. Sequences start at `_00` and are contiguous (`_00, _01,
  _02, ...` — no gaps).
- **Derived group:** if an asset name ends with `_NN`, its style group
  is the name with that suffix stripped. All frames of one animation
  share one group and therefore one visual brief.
- **Sprite size:** each side 1..240 pixels.
- **Sound duration:** 1..2000 ms.

These rules mirror `cube_asset-builder`'s validator. Any prompt that
would cause a downstream validator failure is rejected at the prompter
stage.

## 8. Assumptions

List every design decision made that was not spelled out in the GDD.
Each item: "assumption — rationale". If none, write "All visual
decisions trace directly to the GDD."
```

### Step 6: Write per-asset prompts

For each sprite in the manifest, create
`plans/<game>/prompts/<name>.md`. Every file must be self-contained so
a fresh `canvas-design` invocation can run it without reading the
manifest directly.

Sprite prompt template:

```markdown
# Asset: <name>

## Output contract
- File: `assets/art/<name>.png`
- Size: <W>×<H> pixels, RGBA
- Pivot: (<px>, <py>) — see art bible for how to align artwork to pivot
- Group: `<derived_group>`
- Animation: `<anim>` frame `<N>` of `<total_in_group>` (omit if not animated)

## Visual brief
<One sentence lifted from the manifest's `description` field.>

## Shared style (MUST read first)
See [../art_bible.md](../art_bible.md) AND [../style_profile.md](../style_profile.md).
Apply the palette, line weight, shading policy, abstraction level,
typography, and mood verbatim. Do not invent new style parameters.

## Style-profile reference language (REQUIRED — copy verbatim into your render thinking)
Quote 2–4 sentences from the style profile's "Reference language for
prompts" section. These are the exact phrases the renderer must internalize
before drawing. Example for `detailed_pixelart`:
> "4-tone pixel-art shading: highlight, base, mid-shadow, deep-shadow.
> 1-pixel hard outline using the base ramp's deepest tone.
> Top-left ~35° light, highlight cluster on upper-left, shadow on lower-right."

## Object specifics
<5–10 sentences describing THIS object concretely. Required granularity:
- silhouette and overall shape
- specific feature placement (e.g., for a strawberry: 5-tipped green
  calyx top, 6–8 yellow seed-pip dots, capsule highlight at upper-left)
- per-feature highlight pixels (where individual bright pixels land)
- relationship to neighboring assets in the same group
- motion or pose for this specific frame if animated
The level of detail must match the style profile's detail-density policy
(e.g., `detailed_pixelart` requires 4–8 distinct features per 48×48
sprite, while `minimalist_flat` requires just 1–2).>

## For animation frames only
<Only include when `anim` is set.>
- Sequence role: frame <N> of <total>. Describe what changes from
  frame <N-1> (omit for N=0).
- Pivot stability: the pivot point must remain at (<px>, <py>) across
  all frames of `<anim>` so the engine can place the sprite
  consistently.
- Silhouette continuity: no introduction of visual elements that are
  absent in adjacent frames.

## Naming contract
Save the rendered file as exactly `<name>.png`. Do not change case,
do not add prefixes, do not change extension. The downstream packer
keys everything off this filename.

## Hard no's
- No text other than the brand typeface specified in the art bible.
- No drop shadows that exceed the bounding box.
- No elements that touch the edge of the canvas (respect a 1-pixel
  safety margin on each side).
```

For each sound in the manifest, create
`plans/<game>/prompts/<name>.md` with an analogous template. Sounds are
NOT rendered by `canvas-design` — the sound prompt file exists so that
when an audio renderer is added, the contract is already in place.

Sound prompt template:

```markdown
# Asset: <name> (sound)

## Output contract
- File: `assets/mp3/<name>.mp3`
- Duration: <ms> ms (hard cap 2000 ms)
- Event type: `<pickup|hit|ui|ambient|music|default>`
- Group: `<group>`

## Sonic brief
<One sentence from manifest `description`.>

## Tone & reference
<2–3 sentences: mood, instrument family if any, analogs from GDD §6.>

## Naming contract
Save as exactly `<name>.mp3`. 96 kbps is the packer's expected bit
rate.

## Hard no's
- No clipping.
- No silence longer than 100 ms at the head or tail.
```

### Step 7: Self-check before hand-off

Before reporting success, re-walk the generated prompts and confirm:

1. Every sprite in the manifest has a corresponding
   `prompts/<name>.md`. Every sound has one too. Count matches
   manifest count.
2. Every prompt's `<name>` in the header equals the manifest entry's
   `name` byte-for-byte.
3. Every prompt file links to `../art_bible.md` AND `../style_profile.md`.
4. `asset_index.md` links resolve to existing files.
5. `art_bible.md` §0 "Style Profile (binding)" is present, names the
   chosen preset, and the binding rules block has 3–5 bullets.
6. `art_bible.md` §7 "Asset Naming Contract" is present and
   complete (not truncated or paraphrased).
7. No prompt file references a color, typeface, or style detail that
   is absent from the art bible OR the style profile.
8. Every sprite prompt's "Style-profile reference language" section
   contains 2–4 quoted sentences from the style profile.
9. Every sprite prompt's "Object specifics" section meets the detail-density
   policy of the chosen style profile (count features and per-feature
   highlights; reject thin descriptions for `detailed_pixelart` /
   `retro_8bit` / `painterly_storybook`).

If any check fails, fix in place and re-check. Do not report success
on a partially-generated set.

### Step 8: Hand-off

Print:

> Asset prompts ready.
> - Art bible: `plans/<game>/art_bible.md`
> - Index: `plans/<game>/asset_index.md`
> - Per-asset prompts: `plans/<game>/prompts/*.md` (N sprites + M sounds)
> Typography is locked to `skills/canvas-design/canvas-fonts/Rubik-Bold.ttf`.
> Palette is derived from GDD §1.2 — review the art bible before rendering.
> Next: run `canvas-design` once per `prompts/<name>.md` to produce
> `assets/art/<name>.png`. Sounds are emitted as prompt files but
> wait for a future audio renderer.

Do NOT invoke `canvas-design` automatically. The user decides when
to render and may want to hand-edit the art bible first.

## Constraints

- **Never render pixels.** This skill only writes markdown. Rendering
  is `canvas-design`'s job.
- **Never invent asset names.** Every prompt corresponds to a manifest
  entry. If the manifest is wrong, stop and send the user back to
  `technical_prompter`.
- **Never edit the manifest.** Read-only here.
- **Never edit the GDD.** Read-only here.
- **Never skip validation.** The Step 2 pre-flight is mandatory.
- **One typeface, forever.** `Rubik-Bold.ttf` is the only permitted
  typeface. This is enforced in the art bible and repeated in every
  sprite prompt that contains text.
- **Palette from GDD only.** Do not hard-code a palette in this skill.
  If the GDD is silent, pick values and mark them `(assumed)` in the
  art bible so the user can override.

## Error Handling

| Symptom | Action |
|---------|--------|
| Manifest validation fails | Print all violations; stop. Ask user to fix the manifest (possibly via `technical_prompter`). |
| GDD §1.2 missing or empty | Stop. Ask user to flesh out Style & References in the GDD. |
| Manifest references an animation with non-contiguous frames | List the animation name and the frame gap; stop. |
| A prompt file would collide with an existing hand-edited one | Show a diff, ask user whether to overwrite or skip. Never silently overwrite hand-edited prompts. |
| Rubik-Bold.ttf is not at the expected path | Stop. Ask user to add it before proceeding. |
